# vibe-retirement: 은퇴현금 서비스 문서

## 1. 서비스 개요

본 문서는 40세에서 60세 은퇴 예정자들이 안정적인 노후를 준비하고, 은퇴 후에도 재정적 자유를 누릴 수 있도록 돕는 종합 은퇴현금 설계 서비스인 `vibe-retirement`에 대한 상세한 기술 및 기능 문서를 제공합니다. 이 서비스는 건강보험료 부담 최소화, ISA 계좌 활용을 통한 종합소득세 절세, 퇴직연금 및 개인연금 ETF 투자 전략을 포함하여 실질적인 재정 솔루션을 제시합니다.

### 1.1. 서비스 목적

은퇴 예정자들이 복잡한 금융 및 세무 정보를 쉽게 이해하고, 개인의 상황에 맞는 최적화된 은퇴 계획을 수립 및 실행할 수 있도록 지원합니다.

### 1.2. 서비스 대상

*   **주요 타겟:** 40세 ~ 60세 은퇴 예정자
*   **세부 대상:**
    *   은퇴 후 재정 계획에 대한 막연한 불안감을 가진 사람
    *   건강보험료 등 은퇴 후 고정 지출 관리에 관심이 많은 사람
    *   ISA 계좌를 활용한 절세 전략에 대한 정보가 필요한 사람
    *   퇴직연금, 개인연금 등 연금 자산을 효율적으로 운용하고 싶은 사람
    *   ETF 투자를 통해 은퇴 자산을 증식하고 싶은 사람

### 1.3. 서비스 핵심 가치

*   **맞춤형 솔루션:** 개인의 재정 상황과 은퇴 목표에 맞는 최적화된 은퇴현금 설계 제공
*   **절세 극대화:** 건강보험료, 종합소득세 등 은퇴 후 발생할 수 있는 세금 부담을 최소화하는 전략 제시
*   **수익률 제고:** 퇴직연금, 개인연금, ETF 등 다양한 투자 상품을 활용한 자산 증식 방안 제안
*   **쉬운 이해:** 복잡한 금융 지식을 일반인도 쉽게 이해할 수 있도록 명확하고 직관적인 정보 제공
*   **지속적인 관리:** 은퇴 후에도 변화하는 재정 환경에 맞춰 지속적인 포트폴리오 조정 및 관리 지원

## 2. 주요 기능

`vibe-retirement` 서비스는 다음과 같은 주요 기능을 제공합니다. 각 기능에 대한 상세 내용은 별도 문서에서 다룹니다.

*   **F-01: 개인별 은퇴 목표 및 현재 재정 상태 입력**
*   **F-02: 건강보험료 예상 시뮬레이션 및 최소화 방안 제시**
*   **F-03: ISA 계좌 개설 및 운용 가이드 제공**
*   **F-04: ISA 계좌를 통한 종합소득세 절세 효과 분석**
*   **F-05: 퇴직연금(DC/DB/IRP) 현황 분석 및 최적화 전략 제안**
*   **F-06: 개인연금(연금저축/연금보험) 포트폴리오 진단 및 개선 방안 제시**
*   **F-07: 은퇴 자산 운용을 위한 ETF 추천 및 포트폴리오 구성**
*   **F-08: 투자 성향 분석을 통한 맞춤형 ETF 투자 전략 수립**
*   **F-09: 은퇴 후 현금 흐름 시뮬레이션 (월별 수입/지출 예상)**
*   **F-10: 은퇴 자산 고갈 시점 예측 및 대비책 마련**
*   **F-11: 은퇴 계획 진행 상황 대시보드 제공**
*   **F-12: 전문가 상담 연결 기능**
*   **F-13: 은퇴 관련 최신 세법 및 금융 상품 정보 제공**
*   **F-14: 은퇴 준비 체크리스트 및 목표 달성 알림**

## 3. 서비스 구현 전략

`vibe-retirement` 서비스는 사용자 경험을 최우선으로 고려하며, 다음과 같은 전략을 통해 구현됩니다.

### 3.1. PainPoint 해결

*   **복잡한 정보의 단순화:** 은퇴 재정 관련 복잡한 법규 및 금융 상품 정보를 사용자 친화적인 언어와 시각 자료로 제공하여 진입 장벽을 낮춥니다.
*   **개인화된 가이드:** 일반적인 정보 제공을 넘어, 개인의 상황에 맞는 맞춤형 솔루션을 제시하여 '뭐부터 해야 할지 막막하다'는 PainPoint를 해결합니다.
*   **실행 가능한 전략:** 단순히 정보만 나열하는 것이 아니라, ISA 계좌 개설, ETF 투자 등 구체적이고 실행 가능한 단계를 안내하여 사용자가 직접 재정 계획을 실행할 수 있도록 돕습니다.

### 3.2. WowPoint 강화

*   **즉각적인 피드백:** 사용자가 입력한 정보에 따라 건강보험료 절감액, 예상 세금 절감액, 투자 수익률 등을 즉각적으로 시뮬레이션하여 긍정적인 경험을 제공합니다.
*   **직관적인 시각화:** 은퇴 자산 현황, 현금 흐름, 투자 포트폴리오 등을 직관적인 그래프와 차트로 시각화하여 이해도를 높입니다.
*   **전문성 및 신뢰성:** 최신 세법 및 금융 시장 정보를 반영한 정확한 데이터를 제공하고, 필요한 경우 전문가 연결을 통해 서비스의 신뢰성을 높입니다.

## 6. 중요 고려사항 및 개선 방향

심층 분석 보고서에서 식별된 주요 이슈들을 바탕으로, `vibe-retirement` 서비스의 문서화 및 구현 시 다음 사항들을 중요하게 고려하고 개선해야 합니다.

### 6.1. 규제 준수 및 적합성 프로세스 ([I-04] Critical, P0)

F-07 (ETF 추천) 및 F-12 (전문가 상담) 기능은 단순 정보 제공을 넘어 실질적인 재무/투자/법률 의사결정을 유도할 수 있으므로, 금융 관련 법규 및 규제 준수 체계를 명확히 수립해야 합니다. 적합성/적정성 평가 기준, 투자자 보호 문구, 자문과 정보 제공의 경계, 설명의무, 책임 한계, 기록 보존, 이해상충 공개, 전문가 자격 검증 및 갱신 요건 등을 문서화하고 시스템에 반영해야 합니다.

### 6.2. 민감 재무정보 및 상담이력 보안/프라이버시 ([I-01] High, P0)

사용자의 소득, 자산, 부채, 연금 현황, 상담 이력 등 민감한 재무정보를 다루므로, 데이터 수집 동의 범위, 최소 수집 원칙, 저장/전송 암호화, 접근 제어, 보관 기간, 파기 정책, 제3자 제공 정책, 데이터 주체 요청 처리 절차 등을 명확히 정의해야 합니다. DFD (Data Flow Diagram) 및 권한 매트릭스를 함께 작성하여 보안 및 프라이버시 요건을 충족해야 합니다.

### 6.3. 
실시간 최신 세법/금융정보 출처·검증·갱신 정책 부재 ([I-03] High, P0)

F-13 (최신 세법/금융 상품 정보 제공) 기능의 신뢰성 확보를 위해 정보의 출처 기관, 수집 주기, 검증 프로세스, 게시 전 승인 절차, 게시 실패 시 fallback 정책, '기준일/as-of timestamp' 표기 정책, 오래된 정보에 대한 경고 표시 등을 명확히 문서화해야 합니다. 금융/세무 정보는 시간 민감도가 높으므로, 잘못된 정보 제공으로 인한 사용자 오인을 방지해야 합니다.

### 6.4. 안정형 포트폴리오에 인버스 ETF 포함 오류 수정 ([I-05] High, P1)

F-07 (ETF 추천) 기능에서 '안정형' 투자 성향에 인버스 ETF가 포함된 것은 명백한 요구사항 오류입니다. 인버스 ETF는 변동성이 높은 상품이므로, 안정형 포트폴리오 예시에서 제거하고, 허용 상품군을 보수형/중립형/공격형으로 재정의해야 합니다. 또한 공격투자형에서 언급된 레버리지/인버스 ETF와 연결하여 '고위험 단기 전술 상품'으로 별도 분류하여 문서의 일관성을 유지해야 합니다.

### 6.5. 시뮬레이션 계산 가정·산식·버전·설명가능성 ([I-06] High, P1)

F-09 (현금 흐름 시뮬레이션), F-10 (자산 고갈 시점 예측), F-11 (대시보드) 기능의 핵심 가치는 정확한 수치 계산과 예측에 있습니다. 따라서 핵심 산식, 입력 단위, 할인/실질가치 기준, 기본 가정값, 시나리오 파라미터, 버전 번호, 결과 해석 기준 등을 명확히 문서화해야 합니다. 가정 레지스트리, 산식 문서, 예제 입력/출력, 변경 이력, 결과 화면에 표시할 설명 텍스트, edge case 규칙 등을 정의하여 계산 결과의 신뢰성과 설명 가능성을 확보해야 합니다.

### 6.6. API 계약·데이터 스키마·이벤트 흐름 정의 ([I-07] Medium, P1)

기능 간의 명확한 연동을 위해 API 계약, 데이터 스키마, 이벤트 흐름을 정의해야 합니다. Canonical user financial profile schema, simulation input/output schema, dashboard aggregate schema, alert event schema, audit event schema 등을 정의하고, 각 기능 문서 끝에 '입력', '출력', '참조 소스', '에러 케이스' 섹션을 추가해야 합니다. OpenAPI 또는 JSON Schema 초안을 저장소에 포함하는 것을 권장합니다.

### 6.7. 역할/권한/RBAC 및 최소권한 모델 ([I-02] High, P1)

사용자, 전문가, 관리자 간의 역할과 권한 경계를 명확히 정의해야 합니다. 각 역할별 데이터 접근 범위, 상담 이력 접근 범위, 대시보드 필드 마스킹, 알림 발송 권한, 감사 로그 생성 의무 등을 표 형태로 추가하고, API 설계 전에 RBAC 매트릭스를 확정해야 합니다. 이는 민감 데이터에 대한 접근 통제를 강화하고 과도한 권한 부여로 인한 보안 위험을 방지합니다.

### 6.8. 개발/운영/배포/의존성 문서 ([I-08] Medium, P2)

실제 기술 스택, 버전 명세, 로컬 실행 절차, 환경 변수 목록, DB 마이그레이션 방법, 배포 절차, 운영 체크리스트, 로그/메트릭/알림 기준, 시크릿 관리 원칙 등을 문서화해야 합니다. README에는 Quick Start를, 별도 `docs/ops.md`에는 운영 문서를 두는 구성이 적합합니다.

### 6.9. 테스트 전략·검증 데이터·CI/CD 품질 게이트 ([I-09] Medium, P2)

계산형 서비스의 특성을 고려하여 기능별 acceptance test, 대표 사용자 프로필 세트, 세법 버전별 regression fixture, 계산 결과 snapshot, 콘텐츠 게시 전 검수 체크리스트, merge 차단용 CI 규칙 등을 문서화해야 합니다. 최소한 `lint + schema validation + calculation regression + docs link check` 정도의 CI 규칙을 설정하는 것이 바람직합니다.

### 6.10. 대시보드·시뮬레이션·실시간 정보처리의 성능/운영 설계 ([I-10] Medium, P2)

대시보드 fan-in, 잦은 재계산, 외부 정보 갱신, 알림 큐 처리 등 운영 부담을 고려하여 캐시, 비동기 배치, 선계산(materialization), 재계산 트리거, rate limiting, 실패 재시도, degraded mode 정책 등을 문서화해야 합니다. 각 기능에 허용 응답시간과 stale 허용 범위를 정의하여 성능 및 운영 안정성을 확보해야 합니다.

### 6.11. 시각화 중심 서비스의 접근성 요구사항 ([I-11] Low, P3)

README에서 강조하는 직관적인 시각화와 쉬운 이해를 위해 WCAG 기준 준수 목표, 차트의 텍스트 대안, 키보드 탐색 가능 여부, 색만으로 상태를 전달하지 않는 규칙, 숫자 요약 카드의 스크린 리더 친화 문구, 알림 빈도와 시각적 강조 기준 등을 추가하여 접근성을 강화해야 합니다.

### 6.12. 요구사항 관리·문서 거버넌스·라이선스/출처 정책 ([I-12] Medium, P2)

중복된 문서를 정리하고, 각 기능 문서에 `acceptance criteria`, `non-functional requirements`, `owner`, `last updated`, `depends on`, `related API/schema`, `out of scope`를 추가해야 합니다. 또한 저장소 루트에 LICENSE와 출처/차용 정책을 명시하고, README에는 외부 참고 문서와 자체 산출물의 경계를 선명하게 표시하여 문서의 추적성과 거버넌스를 확립해야 합니다.
